## 一、测试背景

实物调试中观察到 LegBot 在 KEY3 / LOCOMOTION 接管后，`gait=0` 时没有表现出预期的稳定原地对角踏步。由于实物调试同时受到 DDS 通信、电机零位、IMU、接触状态、安全保护和真实动力学影响，单独从实物现象难以判断问题来源。

因此新增一条 MuJoCo 仿真测试路径，用于回答以下问题：

```text
1. 实物 bridge 中的 KEY3 逻辑放到仿真里是否能跑通？
2. 控制器是否真的进入了 gait0？
3. gait0 不明显踏步是 gait 没进，还是进入后姿态/高度/接管状态不对？
4. 启动阶段往上站立和摇晃是否来自站立高度目标不一致？
```

本次仿真不是替代实物测试，而是用于隔离控制链路问题。仿真中如果能复现实物中的“先往上站一下、摇晃、然后进入踏步”的现象，说明问题更可能在控制流程、目标高度或接管姿态，而不是 DDS 通信本身。

## 二、测试问题

### 2.1 最初问题

实物上 `gait=0` 时没有进入预期的稳定原地踏步。需要确认：

```text
gait0 是否没有被控制器接收？
还是 gait0 已经进入，但因为接管姿态/高度不对，导致视觉上不像稳定踏步？
```

### 2.2 进一步问题

仿真启动时从趴下姿态开始，通过 Kp/Kd 插值站立后进入 KEY3 逻辑。实际观察到：

```text
1. 插值站立后姿态不完全正确；
2. 进入 LOCOMOTION 后身体会继续往上站一下；
3. 随后发生一定摇晃；
4. 稳定一段时间后进入 gait0 踏步；
5. gait0 对角接触切换存在，但踏步效果不够理想。
```

因此需要重点分析：

```text
站立阶段 base_z 为什么没有达到预期？
MPC/WBC 接管后的 target_z 是否与站立阶段目标一致？
gait0 的接触相位和摆腿是否真的开始切换？
```

## 三、仿真验证方案

### 3.1 进程拓扑

本次验证使用两个终端：

```text
终端 1：MuJoCo LegBot dryrun 仿真桥
        mujoco_legbot_key3_dryrun_sim
        ↓
        SharedMemory development-simulator
        ↑
终端 2：MIT_Controller 仿真模式
        mit_ctrl v s
```

数据流：

```text
MuJoCo model/data
  ↓
fillRobotInputs()
  ↓
SimulatorSyncronizedMessage / SharedMemory
  ↓
SimulationBridge
  ↓
RobotRunner
  ↓
MIT_Controller / ControlFSM / ConvexMPC / WBC
  ↓
SpiCommand
  ↓
applyControllerOutput()
  ↓
MuJoCo actuator ctrl
```

### 3.2 控制时序

仿真桥支持从趴下姿态启动：

```text
0s - standup_ramp 结束：
  q-only Kp/Kd 从 prone/down 姿态插值到 stand 姿态

standup_ramp - locomotion_start：
  q-only Kp/Kd 保持 stand 姿态

locomotion_start：
  请求 control_mode = LOCOMOTION
  请求 cmpc_gait = 4
  即 gait4 standing

trot_start：
  请求 cmpc_gait = 0
  即 LegBot stationary slow trot

forward_start：
  可选，请求 forward gait 和速度
```

本轮为了隔离 gait0，后续测试取消前进，只保留原地踏步：

```text
start-prone -> Kp/Kd standup -> gait4 standing -> gait0 stationary trot
```

## 六、测试现象与日志结论

### 6.1 gait0 实际已经进入

仿真终端出现：

```text
[MuJoCo LegBot Key3 Dryrun] Requesting gait0 stationary trot at t=13.001
[KEY3_DRYRUN_SIM] phase=GAIT0_DIRECT_SIM requested_gait=0
```

控制器终端对应出现：

```text
Response set user double cmpc_gait user to: 0
```

因此当前问题不能简单描述为“没有进入 gait0”。更准确的描述是：

```text
控制器已经进入 gait0；
但进入 gait0 前姿态和高度准备不足；
gait0 切入瞬间 qerr 较大，导致踏步观感和稳定性不理想。
```

### 6.2 gait0 对角支撑在切换

gait0 后仿真接触力出现对角交替：

```text
Fn=[1.79 43.25 79.04 2.81]
Fn=[0.00 68.16 72.82 0.00]
Fn=[63.15 0.00 0.00 71.58]
Fn=[0.00 72.37 69.95 0.00]
```

这说明 gait0 的对角相位和支撑切换已经发生，不是 gait scheduler 停在 standing。

### 6.3 Kp/Kd 站立阶段没有追到目标

站立保持阶段日志：

```text
STANDUP_HOLD base_z=0.24172
qerr_norm=0.398
q_target=[-0.040 0.936 -1.779 ...]
```

实际关节仍接近：

```text
FR q=[-0.017 0.866 -1.931]
RR q=[-0.036 0.896 -2.001]
```

说明 q-only Kp/Kd 站立阶段没有真正追到目标关节角，机身实际高度停在约 `0.242m`。

### 6.4 进入 LOCOMOTION 后出现向上拉高

控制器原先目标高度：

```text
target_z=0.277000
```

而站立阶段实际高度：

```text
base_z≈0.2417
```

进入 gait4 后，控制器把机身拉到：

```text
body_z≈0.272 - 0.280
```

这与视觉观察中的“先往上站一下，然后摇晃”一致。

### 6.5 gait0 切入瞬间误差较大

gait0 切入第一帧：

```text
phase=GAIT0_DIRECT_SIM
qerr_norm=0.52306
max_qd_des=0.8968
```

随后 qerr 会下降，但第一下切入已经会引入明显姿态扰动。

## 七、原因分析

### 7.1 站立阶段低不是 MPC 优化权重直接导致

在 `start_prone && !locomotionRequested` 阶段，仿真桥执行的是纯关节 PD：

```cpp
ctrl = kp * (q_des - q) - kd * qd;
```

此阶段没有 MPC、QP-WBC，也没有直接的 `base_z` 优化目标。因此：

```text
STANDUP_HOLD 阶段 base_z 偏低，不能首先归因于 MPC base_z 权重。
```

更直接的原因是：

```text
1. q-only PD 没有追到目标 q；
2. 腿部实际姿态比目标更蜷缩；
3. 接触、摩擦、重力和关节力矩平衡后，机身高度自然停在更低位置；
4. Kp/Kd、站立目标、插值时间、力矩限制都会影响这个平衡。
```

### 7.2 MPC/WBC 接管后的高度目标不一致会放大摇晃

站立阶段实际高度约：

```text
base_z≈0.242
```

控制器原先接管目标：

```text
target_z=0.277
```

两者相差约：

```text
0.277 - 0.242 = 0.035m
```

也就是约 `3.5cm`。对于小型四足机器人，这个高度跳变已经足以造成明显的向上拉高和姿态扰动。

因此将控制器 LegBot nominal height 临时改为 `0.260`，用于验证：

```text
如果 target_z 降低后，进入 LOCOMOTION 时向上拉高和摇晃明显减轻，
则说明高度目标不一致是主要原因之一。
```

### 7.3 gait0 不是前进步态

当前 LegBot 逻辑中：

```text
cmpc_gait = 4：standing
cmpc_gait = 0：stationary slow trot
```

对 LegBot 的 `gait0`，`ConvexMPCLocomotion::_SetupCommand` 中会将速度命令置零，因此它用于原地对角踏步，不用于前进。如果需要前进，应在 gait0 原地踏步稳定后切换到其他 gait，例如 `gait=5`，并给定 `--speed 0.08`。

### 7.4 当前仿真复现了实物现象

仿真中观察到：

```text
启动阶段往上站一下
随后有摇晃
之后进入 gait0 踏步
```

这与实物观察一致，说明本仿真验证路径可以用于继续排查实物问题。当前更倾向于认为问题主要在：

```text
站立准备姿态不足
高度目标切换不连续
gait4 -> gait0 切换瞬间 qerr 较大
```

而不是 DDS 通信层没有发送 gait0，或 gait scheduler 完全没有进入 gait0。

## 八、当前结论

### 8.1 已确认

```text
1. 新 MuJoCo dryrun 仿真桥可以复现实物 KEY3 类似流程。
2. mit_ctrl v s 可以与新仿真桥通过 SharedMemory 正常闭环。
3. gait0 请求已被控制器接收。
4. gait0 的对角接触力和支撑相位确实在切换。
5. 原先进入 LOCOMOTION 后 target_z=0.277，会把站立阶段较低的机身继续往上拉。
6. 启动摇晃与站立阶段 qerr 未收敛、接管目标高度不一致高度相关。
```

### 8.2 尚未完全确认

```text
1. 将 LegBot 控制器目标高度改成 0.260 后，摇晃能降低多少。
2. q-only 站立阶段 base_z 只能到 0.242 的根本原因是 Kp/Kd 不足、目标姿态不合适，还是 MuJoCo 接触/力矩限制造成。
3. 实物中 gait0 视觉上不明显踏步，是否与仿真中同一个 qerr / height handoff 问题完全一致。
4. gait0 稳定后切 gait5 前进时，是否还存在 WBC force 瞬时掉零或对角支撑冲击。
```

### 8.3 当前最重要判断

```text
“gait0 没进入”不是当前最准确的结论。

更准确的结论是：
gait0 已进入，并且对角支撑在切换；
但进入 gait0 前机器人没有处在一个足够稳定、足够接近目标高度和目标关节姿态的状态，
导致切换瞬间扰动较大，踏步效果不理想。
```

## 九、后续验证建议

### 9.1 验证 0.260 控制高度

重新运行只踏步测试，重点检查控制器终端：

```text
target_z=0.260000
```

观察：

```text
1. gait4 接管后是否仍明显向上拉高；
2. body_z 是否稳定在 0.260 附近；
3. qerr_norm 是否小于原先 gait0 切入时的 0.523；
4. gait0 对角踏步是否更平顺。
```

### 9.2 改进 q-only 站立阶段

如果 `target_z=0.260` 后仍然摇晃，应重点调试 Kp/Kd 站立阶段：

```text
1. 增大 standup_kp，例如 50 / 70 / 80；
2. 适当增大 standup_kd，例如 5 / 6；
3. 延长 standup_ramp 和 standup_hold；
4. 打印每条腿 q error，找出主要误差来自 thigh 还是 calf；
5. 增加站立收敛条件：base_z 和 qerr 达标后才允许进入 LOCOMOTION。
```

建议收敛标准：

```text
base_z >= 0.255
qerr_norm < 0.10
roll/pitch 较小
四足接触力无明显偏载
```

### 9.3 加入自动等待接管条件

当前进入 LOCOMOTION 是时间型：

```text
data->time >= locomotion_start
```

可改为：

```text
时间达到
并且 base_z / qerr_norm / 姿态角满足阈值
再请求 gait4
```

这样可以避免机器人还没有站稳就把控制权交给 MPC/WBC。

### 9.4 分离 gait0 与前进 gait

建议分两组测试：

```text
A. start-prone -> gait4 -> gait0
   只验证原地踏步是否稳定

B. start-prone -> gait4 -> gait0 -> gait5 + speed 0.08
   在 A 稳定后再验证前进
```

不要在 gait0 尚未稳定前直接判断前进能力，否则容易把站立接管问题和前进步态问题混在一起。

### 9.5 实物侧对应检查

实物 bridge 中建议重点对比：

```text
1. q-only standup 结束时真实 q 与 StandQDds() 的误差；
2. 实物估计 base_z 或等效腿长是否接近 0.260；
3. key3 进入 gait4 direct 前 qerr 是否足够小；
4. gait0 shadow 第一帧 q_des / qd_des 是否发生过大跳变；
5. RuntimeSafetyGuard 是否因为姿态、dq 或 q_des jump 介入。
```
